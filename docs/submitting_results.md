# Submitting Results

This project is designed for reproducible, defensive evaluation results.

## Result Requirements

Public result submissions should include:

- Dataset version from `DATASET_VERSION`.
- Repository commit SHA.
- Model or system label.
- Run config matching `evaluation/run_config.schema.json`.
- Prompting and structured-output method.
- Model output JSONL or a safe public summary.
- Score report generated by `scripts/score_model_outputs.py`.
- Notes on any safety flags, skipped cases, or manual review.

Do not publish private prompts, proprietary logs, secrets, personal data, or harmful operational details.

## Suggested Workflow

Generate prompts:

```bash
python scripts/make_prompt_batch.py data/sample_alerts.jsonl --output .tmp/prompt_batch.jsonl
```

Run your model or system against the generated prompts and write output records matching `evaluation/output_schema.json`.

Validate the run config:

```bash
python scripts/validate_run_config.py path/to/run_config.json
```

Validate outputs:

```bash
python scripts/validate_model_outputs.py data/sample_alerts.jsonl path/to/model_outputs.jsonl
```

Score outputs:

```bash
python scripts/score_model_outputs.py data/sample_alerts.jsonl path/to/model_outputs.jsonl --markdown path/to/score_report.md
```

Summarize the run using `docs/result_card_template.md`.

## Interpreting Scores

The automated score is a reference aid. It is useful for consistency checks, regression tracking, and identifying cases for deeper analyst review. It should not be treated as a production-readiness claim.

Grouped scores by category, difficulty, and failure mode are often more useful than a single average. They can highlight whether a model struggles with missing context, over-escalation, under-escalation, or safe action scoping.
